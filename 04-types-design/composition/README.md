## Why This Section Matters

Go favors explicit composition over inheritance-heavy class hierarchies.
That means learners need to be able to answer three questions clearly:

- when should a type own another type as a named field?
- when does embedding make access cleaner without hiding too much?
- when is shadowing behavior helpful, and when does it make the model harder to read?

Use `CO.1` and `CO.2` to build those answers before taking on `CO.3`.
